Firecrawl's /extract Deprecation Resets the Extraction API Conversation

Moving from /extract (URL-in, schema-out) to /agent (intent-in, results-out) is a paradigm shift. Developers building integrations today learn the /agent mental model first — the AI decides where to look. SDK extract() methods are deprecated across JS and Python, so any tutorial referencing /extract is a dead end. For Novada, this means the Extract endpoint (pending infra activation) must be positioned deliberately: either as a structured, deterministic complement to /agent, or redesigned as agent-first. Shipping URL-in/schema-out in May 2026 without a clear narrative is a positioning risk, not a gap.

Firecrawl /extract 废弃重置了提取 API 对话

从 /extract（URL 输入/Schema 输出）到 /agent（意图输入/结果输出）是范式转变，而非简单的更名。今天构建集成的开发者首先学习 /agent 思维模型——AI 决定去哪里查找。JS 和 Python SDK 的 extract() 方法均已废弃，任何引用 /extract 的教程都是死路。对 Novada 而言，这意味着待激活的 Extract 端点必须有意识地定位：要么作为 /agent 的结构化、确定性补充，要么重新设计为 Agent 优先。在 2026 年 5 月发布无明确叙事的 URL 输入/Schema 输出是定位风险，而非功能缺口。
